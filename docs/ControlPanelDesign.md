# Lucid Engine IDE: Bottom Panel Design

To build a professional, modern IDE (similar to VS Code and Unreal Engine), the bottom dock of the window is critical. It serves as the primary feedback loop for the developer. 

Here is the proposed design for the Bottom Panel, broken down by the exact tabs you need.

---

## 1. Terminal (The OS Shell)
This is a raw instance of the user's operating system terminal (PowerShell on Windows, Bash/Zsh on Mac/Linux) embedded directly into the UI using a library like `xterm.js`.
* **Purpose:** Running external tools.
* **What developers type here:** `git status`, `luc build --release`, `npm install`.
* **Design Features:**
    * Support for multiple terminal instances (Terminal 1, Terminal 2).
    * Standard ANSI color support.

## 2. Output (The Read-Only Logs)
This tab catches all the text that the C++ Kernel, the Asset Baker, and the running Game print to `stdout` and `stderr`.
* **Purpose:** Monitoring engine health and background processes.
* **Design Features:**
    * **Filter Buttons:** Toggles for `[Errors]`, `[Warnings]`, and `[Info]`.
    * **Category Dropdown:** Let the user filter by engine subsystem (e.g., only show `[Audio]` or `[Physics]` logs).
    * **Clickable Links:** If a log prints `Error in player.luc at line 42`, clicking it should instantly open that file in the main code editor.

## 3. Problems (The Compiler Diagnostics)
Instead of forcing developers to read raw text to find errors, this tab provides a clean, organized table of every bug currently in their code.
* **Purpose:** A clean interface for the Luc compiler's output.
* **Design Features:**
    * Powered by the `luc_langserver` (LSP). It updates in real-time as the developer types.
    * Columns: `Severity (Icon) | Message | File | Line`.

## 4. Engine Console (The "Commands" Section)
This is the tab you were asking about! This is what Unreal Engine uses at the bottom of its Output Log. It is a highly integrated interactive prompt (REPL) that talks directly to the engine's memory.
* **Purpose:** Live modification of the Editor or the running Game.
* **What developers type here:** Luc code or registered C++ commands. E.g., `player.health = 0` or `r.SetResolution 1920 1080` or `SpawnEntity("Orc")`.
* **Design Features:**
    * **IntelliSense Autocomplete:** Because it talks to the engine, if the developer types `player.`, a popup should appear showing `health`, `speed`, and `inventory`.
    * **Two Modes:**
        * *Editor Mode:* Commands affect the Editor itself (e.g., `editor.OpenAsset("hero.prefab")`).
        * *Runtime Mode:* If the game is currently "Playing", commands are sent via IPC (Inter-Process Communication) to the running game instance to affect live gameplay.
    * **History:** Pressing the Up Arrow cycles through previously typed commands.

## 5. The 3-File Console Ecosystem

To make the **Engine Console** work like a professional tool (Unreal/Source), it relies on three specific files working in sync:

### A. The Interpreter (`luc_console.cpp`)
*   **Location:** `kernel/src/core/console_interpreter.cpp`
*   **Role:** The "Brain." It receives the text string from the UI, parses it, and executes the logic.
*   **Logic Flow:**
    1.  **Tokenize:** Split `"player.health = 100"` into chunks.
    2.  **Validate:** Check the `symbols.json` to ensure `player` exists and has a `health` field.
    3.  **Execute:** Write the new value directly into the ECS memory buffer or invoke a registered Luc function.

### B. The Metadata (`symbols.json`)
*   **Location:** `user_projects/GameName/build/symbols.json`
*   **Role:** The "Map." This file is generated by the Luc Compiler.
*   **Problem Solved:** Compiled memory is "blind." This file restores names to memory addresses, allowing for **IntelliSense Autocomplete** in the console.

### C. The Shortcuts (`debug_commands.luc`)
*   **Location:** `user_projects/GameName/src/debug_commands.luc`
*   **Role:** The "Shortcuts." 
*   **Usage:** A standard Luc source file where developers write "Macro" functions like `god_mode()` or `spawn_boss()`. The developer types the function name, and the Interpreter runs the full script logic.

## 6. C++ Header Draft (`console_interpreter.hpp`)

This header defines the interface for the C++ side of the Engine Console. It handles the communication between the UI and the Engine Kernel.

```cpp
#pragma once
#include <string>
#include <vector>
#include <functional>
#include <unordered_map>

namespace Lucid {

    // Represents a command that can be called from the console
    using ConsoleCommandFunc = std::function<void(const std::vector<std::string>& args)>;

    class ConsoleInterpreter {
    public:
        ConsoleInterpreter() = default;

        // Main entry point: Called when a string is received from the IDE UI
        void Execute(const std::string& input);

        // Register a C++ function as a console command (e.g. "quit", "clear")
        void RegisterCommand(const std::string& name, ConsoleCommandFunc func);

        // Autocomplete helper: returns a list of suggestions based on current input
        std::vector<std::string> GetSuggestions(const std::string& partial_input);

    private:
        // Internal command registry (ConCmds)
        std::unordered_map<std::string, ConsoleCommandFunc> m_commands;

        // Helper to parse strings like "player.health = 100"
        struct ParsedCommand {
            std::string target; // e.g. "player"
            std::string field;  // e.g. "health"
            std::string value;  // e.g. "100"
            bool is_assignment;
        };

        ParsedCommand ParseInput(const std::string& input);
        
        // Interaction with the ECS and Symbol Metadata
        void ApplyVariableChange(const ParsedCommand& cmd);
        void InvokeCommand(const std::string& name, const std::vector<std::string>& args);
    };

} // namespace Lucid
```

### Summary of Workflow
If a developer wants to commit their code to GitHub, they click **Terminal**.
If a developer wants to see why their audio isn't loading, they click **Output**.
If a developer has a typo in their code, they look at **Problems**.
If a developer wants to instantly kill all enemies in the level while playing, they click **Engine Console** and type `world.kill_all("enemy")`.
